# План: Нормализация данных в базе данных приложения TaskControl

## Контекст
Пользователь рассматривает возможность нормализации базы данных приложения, вынеся категории и приоритеты задач в отдельные таблицы. Основная цель — улучшить структуру БД, избежать дублирования и повысить гибкость. Однако пользователь обеспокоен значительным объемом изменений в коде, которые потребуются для реализации этого.

## Анализ текущего состояния

### Существующие таблицы

1.  **`tasks`**: Основная таблица для хранения задач. Содержит поля `category` и `priority` как текстовые строки (например, "работа", "высокий").
2.  **`app_state`**: Таблица для хранения глобального состояния приложения (например, флаг инициализации демо-данных). Не связана с `tasks` и не требует изменений.

### Ключевые точки взаимодействия с БД

- **`models.py`**: Определяет `Task` как dataclass. Содержит списки `CATEGORIES`, `PRIORITY_OPTIONS`, `RECURRENCE_OPTIONS`, `STATUS_OPTIONS`, которые используются для валидации.
- **`database.py`**: Класс `DatabaseManager` управляет схемой БД и CRUD-операциями. Метод `_create_schema` создает таблицы. Все методы, работающие с задачами, используют текстовые поля.
- **`services.py`**: Класс `TaskService` содержит бизнес-логику. Метод `_validate_task_data` использует списки из `models.py` для проверки корректности `category` и `priority`. Метод `create_task_from_ai` также использует эти списки.
- **`main.py`**: Инициализирует `DatabaseManager` и `TaskService`.

## Рекомендуемый подход: Пошаговая нормализация

Полная переделка половины кода за один раз рискованна. Вместо этого, я предлагаю пошаговый подход, ��оторый минимизирует риски и позволяет тестировать изменения на каждом этапе.

### Этап 1: Создание новых таблиц и миграция данных

Цель: Создать новые таблицы `categories` и `priorities` и заполнить их данными.

1.  **Модификация `database.py`**:
    - Добавить в метод `_create_schema` создание таблиц `categories` и `priorities`.
    - Заполнить их начальными значениями из `CATEGORIES` и `PRIORITY_OPTIONS`.
    - Обновить индексы для оптимизации поиска.

### Этап 2: Модификация модели и валидации

Цель: Обновить модель `Task` и логику валидации для работы с новыми таблицами.

1.  **Модификация `models.py`**:
    - Удалить списки `CATEGORIES` и `PRIORITY_OPTIONS` из модуля. Их источником теперь будет БД.
    - Добавить методы в `Task` или создать отдельный класс для получения списка допустимых категорий и приоритетов из БД.
2.  **Модификация `database.py`**:
    - Добавить методы `get_all_categories()` и `get_all_priorities()` в `DatabaseManager`.
3.  **Модификация `services.py`**:
    - Обновить метод `_validate_task_data` в `TaskService` для использования новых методов `DatabaseManager` вместо статических списков.

### Этап 3: Обновление CRUD-операций

Цель: Изменить операции чтения и записи задач для использования внешних ключей.

1.  **Модификация `database.py`**:
    - Изменить схему таблицы `tasks`, заменив `category TEXT` на `category_id INTEGER REFERENCES categories(id)` и `priority TEXT` на `priority_id INTEGER REFERENCES priorities(id)`.
    - Обновить все методы (`create_task`, `update_task`, `_row_to_task` и т.д.) для работы с `category_id` и `priority_id` вместо текстовых значений.
    - Добавить методы для преобразования текстовых значений в ID и наоборот.

### Этап 4: Обновление бизнес-логики и UI

Цель: Убедиться, что весь код, использующий категории и приоритеты, работает корректно.

1.  **Модификация `services.py`**:
    - Обновить методы, которые возвращают данные для UI (например, `get_statistics`), чтобы они возвращали текстовые представления, а не ID.
2.  **Модификация `main.py` и UI-компонентов**:
    - Убедиться, что UI-элементы (например, выпадающие списки) получают актуальные списки категорий и приоритетов из БД через `TaskService`.
